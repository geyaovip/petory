# Petory 服务端与管理端路线图

> 管理端版本号使用 **B** 前缀（Backend），与客户端 **1.x** 独立编号。  
> 客户端当前：**V1.4**（本地 Mock，无后台）。  
> **B1.3 已交付**：系统配置 UI、增强仪表盘、审计/登录日志、设备标记、只读运营角色。

## 总览

```
客户端 Mock 阶段 ─────────────────────────────────────────►
                    B1.0        B1.1        B1.2        B1.3        B2.0
                    MVP API     多姿势+码   Kimi代理    运营完善    支付+同步
                         │           │           │           │           │
客户端接入真实后台 ──────┴───────────┴───────────┴───────────┴───────────┘
建议客户端版本      C2.0        C2.1        C2.2        C2.3        C3.0
```

> **C2.x**：建议用于「接入真实后台」的客户端大版本，避免与已发布的 1.4 混淆；具体命名发布前可再定。

## 阶段 0：当前（无后台）

| 项目 | 状态 |
|------|------|
| 客户端版本 | V1.4 |
| 账号 | 本地 `mock-users.json` + 离线模式 |
| 额度 | 本地 `usage-log.json`，日切重置 |
| 兑换码 | 硬编码 `PETORY-PRO-DEMO` 等 |
| MiniMax / Kimi | 客户端 `.env` 或设置中的 Key |
| 运营 | 无后台可查 |

**结论**：可先继续发客户端 1.x 小版本，但 **公开发布正式版前** 应至少完成 **B1.0 + B1.1** 与客户端 C2.x 对接。

## B1.0 — API MVP（后台首版）

**目标**：替代客户端直连 MiniMax，建立真实账号与生成任务闭环。

### 范围

| 模块 | 内容 |
|------|------|
| 工程 | Monorepo 或独立 repo、PostgreSQL、Prisma、对象存储、CI |
| Auth | **假登录**（邮箱 + 密码注册/登录）、JWT、`GET /api/me`（与客户端 Mock 形态一致，数据落库） |
| 设备 | `POST /api/devices/register` |
| 额度 | 生成日额度 3/50，扣减规则对齐 [CLIENT-ALIGNMENT](../../backend/CLIENT-ALIGNMENT.md) |
| 生成 | 单图任务（`styleType` + `pose=idle`）、MiniMax 代理、任务查询 |
| 存储 | 输入图 + MiniMax 原图 URL |
| Admin API | 用户列表/详情、任务列表/详情、禁用用户、加额度、手动 Pro |
| 管理后台 | 最小可用：登录 + 用户 + 任务 + 额度操作 |
| 风控 | IP/用户频率限制、上传大小 |

### 不包含

- 多姿势 batch
- Kimi 代理
- 兑换码后台（仍可用客户端 Mock 码过渡）
- 支付
- 对话额度服务端校验

### 客户端协同（建议 C2.0）

- 登录页仍用邮箱 + 密码，请求改调后台 `POST /api/auth/login`（**不做验证码**）
- 生成流程改调 `POST /api/generation/jobs`
- 移除正式包中的 MiniMax Key
- 额度展示改读 `GET /api/generation/quota`

### 验收

- [ ] 新用户从注册到生成 1 张 idle 图全链路通
- [ ] 额度用尽返回明确错误
- [ ] 管理员可在后台看到任务与失败原因
- [ ] 安全扫描确认 Key 不在客户端

## B1.1 — 生成与商业化对齐

**目标**：与客户端 V1.1–V1.4 已发能力对齐，可下线 Mock 兑换码。

### 范围

| 模块 | 内容 |
|------|------|
| 生成 | **多姿势 batch**（1 次额度扣减，N 个 pose）；6 种 `styleType` |
| 生成 | `pose_completion`、`single_pose_regen` 任务类型（不扣主额度） |
| 权益 | 服务端校验风格 / 姿势 / max_pets / max_desktop_pets |
| 兑换码 | 创建、列表、核销 API + 管理后台页面 |
| Admin | 仪表盘基础指标、兑换码管理 |
| Prompt | 服务端维护 style × pose 模板（从客户端 `posePrompts` 迁移） |

### 客户端协同（建议 C2.1）

- `runGenerationPipeline` 改为 1 次 batch API
- `completePoses` / `regeneratePose` 对接对应 API
- 兑换走 `POST /api/redeem`
- `entitlements` 默认值改由 `GET /api/me` 下发（客户端缓存）

### 验收

- [ ] 免费用户 batch 仅 3 pose，Pro 为 6 pose
- [ ] 姿势补全、单姿势重生成不扣每日 3/50 主额度
- [ ] 兑换码在后台创建后客户端可核销
- [ ] 6 风格 Pro 校验与服务端一致

## B1.2 — 对话代理

**目标**：Kimi Key 迁到服务端，对话额度服务端权威。

### 范围

| 模块 | 内容 |
|------|------|
| Chat API | 代理 Kimi，支持 personality system prompt |
| 额度 | `daily_chat_limit` 20 / 9999，扣减与查询 API |
| 日志 | 对话请求日志（可不存全文，存 metadata） |
| Admin | 用户详情展示今日对话次数、失败率 |
| 风控 | 对话频率限制 |

### 客户端协同（建议 C2.2）

- `chat/handlers.ts` 改调后台
- 设置页移除 Kimi Key 输入（或仅高级调试保留）
- 剩余对话次数展示来自服务端

### 验收

- [ ] 免费用户第 21 次对话被拦截
- [ ] Key 不出现在客户端
- [ ] 后台可统计今日对话量

---

## B1.3 — 运营与配置完善

**目标**：减少改环境变量发版，增强可观测性。

### 范围

| 模块 | 内容 |
|------|------|
| 系统配置 UI | 免费/Pro 额度、超时、维护公告、是否开放注册 |
| 仪表盘 | 成本趋势、失败原因分布、额度不足次数 |
| 日志 | 管理员 audit log、登录日志 |
| 设备 | 设备列表、异常设备标记 |
| 角色 | 可选：只读运营账号 |

### 客户端协同（建议 C2.3）

- 启动时拉取维护公告
- 额度/权益支持后台热更新（短缓存）

---

## B2.0 — 支付与云同步基础

**目标**：商业化闭环 + 换机数据基础。

### 范围

| 模块 | 内容 |
|------|------|
| 支付 | Stripe（或选定供应商）+ Webhook |
| 订阅 | `subscriptions` 全生命周期 |
| 同步 | 宠物元数据上传/拉取（非图片文件或可选购 CDN） |
| 导入导出 | 对接客户端 `schemaVersion: 2` 备份格式 |
| 审核 | 图片审核（生成前或生成后） |

### 客户端协同（建议 C3.0）

- 应用内购买 / 订阅管理页
- 云同步开关（元数据优先）
- 真实登录必选（离线模式策略再定）

---

## B2.x / V2.0 — 长期

| 能力 | 说明 |
|------|------|
| 云端 rembg | 客户端打包成本过高时评估 |
| 生成历史用户页 | 用户查看历史原图 |
| 多设备同步 | 桌宠位置、设置同步 |
| 多语言 | **客户端专项**，后台仅错误码 i18n |
| 开放 API / 市场 | 与初稿 V2.0 一致 |

## 与客户端已发布版本对照

| 客户端版本 | 已发能力 | 依赖后台版本 |
|------------|----------|--------------|
| V1.0 | Mock 登录、额度、更新 | 无（Mock） |
| V1.1 | 6 风格 | B1.1 风格枚举 + 校验 |
| V1.2 | 多宠桌面 | B1.1 权益 `max_desktop_pets` |
| V1.3 | 多姿势、睡觉、姿势补全 | B1.1 batch + completion |
| V1.4 | Alpha 命中、单姿势重生成、音效、成长 | B1.1 `single_pose_regen`；其余纯客户端 |

**建议上线顺序**：不要要求用户先升到 C2.x 再发 1.5 小功能；**功能型 1.x 可继续**，但 **商业正式运营** 前应完成 B1.0–B1.1 与 C2.0–C2.1 对接。
